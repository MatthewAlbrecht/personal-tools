# Playlist Lyrics Design

## Goal

Build a standalone `/playlist-lyrics` tool for curated song collections that are not Genius albums. The first use case is a record-club "Music Tour" playlist, but the feature should support any future playlist-style lyric packet.

The tool should let the user create playlists, paste individual Genius song URLs, scrape each song's metadata and lyrics, add notes, print the finished packet, edit existing playlists, and sync the data to production.

## Non-Goals

- No public playlist lyric pages are required.
- No album crawling for playlist lyrics.
- No required manual entry for song title, artist, or album.
- No MVP-only shortcuts that would prevent reusable scrape storage.

## Route Design

- `/playlist-lyrics`: list page for all playlist lyric packets.
- `/playlist-lyrics/[slug]`: print/read page for a playlist.
- `/playlist-lyrics/[slug]/edit`: edit page for a playlist.

On the list page, clicking a playlist title opens print/read mode. Each row also has an edit pencil that links directly to edit mode. The list page includes a "New Playlist" button that immediately creates an untitled draft and opens the blank editor.

## Data Model

Use a separate playlist system with a shared Genius song scrape cache.

### `playlistLyrics`

Stores playlist-level data:

- `title`
- `slug`
- `theme`
- `description`
- `notes`
- `status`: `draft` or `ready`
- `createdAt`
- `updatedAt`

### `geniusLyricScrapes`

Stores canonical scraped Genius song data:

- `canonicalUrl`
- `songTitle`
- `artistName`
- `albumTitle`
- `lyrics`
- `about`
- `scrapeStatus`
- `lastScrapedAt`
- `createdAt`
- `updatedAt`

`canonicalUrl` is the dedupe key. It is generated by validating a Genius song URL and stripping query string and hash.

### `playlistLyricsItems`

Stores ordered membership and user-authored content:

- `playlistId`
- `lyricScrapeId`
- `position`
- `userNote`
- optional display overrides for `songTitle`, `artistName`, and `albumTitle`
- `pendingUrl` for failed scrape rows that have not produced a `lyricScrapeId`
- `scrapeState`: `scraping`, `ready`, `failed`, or `reused`
- `createdAt`
- `updatedAt`

Scraped metadata should fill the UI by default. Overrides exist only to clean up awkward or incorrect Genius display text.

## Editor Behavior

The editor starts blank for new playlists and saves a draft row immediately with the fallback title `Untitled Playlist`. Editable playlist fields are title, theme, description, and notes. Field edits should save through explicit Convex mutations on blur or debounced change.

Song intake is URL-first:

1. User pastes a Genius song URL.
2. App validates it as a Genius URL.
3. App normalizes it by removing query string and hash.
4. App checks `geniusLyricScrapes` for that canonical URL.
5. If a scrape exists, reuse it and add a playlist item.
6. If no scrape exists, scrape the song immediately, store the result, then add a playlist item.

Each song row should show scrape state: scraping, ready, failed, or reused cached scrape. Failed rows keep the pasted URL in `pendingUrl` and offer retry and delete actions.

The user can reorder songs, delete songs, add per-song notes, and optionally override scraped title/artist/album display fields.

## Print/Read Behavior

The print/read page should match the existing album lyric page behavior:

- show formatted lyrics
- show or hide Genius info/about text
- print normally
- print compact/two-column layout
- preserve the playlist order

The header should use playlist metadata rather than album metadata. It should always show the playlist title and show theme, description, and notes when present.

## Production Sync

The `/playlist-lyrics` list page should include a "Sync to Production" button similar to the existing lyrics tool. Sync should include playlist containers, ordered playlist items, and any referenced Genius scrape rows needed to render the playlists.

The sync action should report created, updated, skipped, and failed counts.

## Error Handling

- Invalid non-Genius URLs should be rejected before scraping.
- Genius fetch or parse failures should leave a failed row with retry/delete actions.
- Duplicate pasted URLs in the same playlist should be prevented.
- Existing scrape reuse should be visible so the user knows the paste worked without a fresh scrape.
- Rescrape should be available per song for stale or incorrect scrape data.

## Testing And Verification

There is no configured test framework in this repository. Implementation should at minimum run:

- `pnpm typecheck`
- `pnpm check`

Focused tests or scripts should verify:

- Genius URL normalization strips query string and hash.
- Existing scrape rows are reused by canonical URL.
- Playlist item order is preserved.
- Optional display overrides do not mutate canonical scrape rows.
- Print/read mode renders from playlist items and scrape rows.
